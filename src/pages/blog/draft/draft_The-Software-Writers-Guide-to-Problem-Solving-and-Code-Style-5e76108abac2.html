What exists already within the software world?

I really like the smalltalk book by Kent Beck. This provides a set of patterns. The patterns are essentially common solutions to common problems. The majority of these are in the micro.

Then there exists a series of high level literature for architecture and code structure. Things like Onion Architecture, Clean Code, Hexagonal Architecture, Micro-services, etc.

There are also workflow guides like Agile and Extreme Programming that revolve around way a team should act to product high quality software.

I still have a gap in my mind that fits together the architecture and the workflow, using common patterns and styles. What tools can we provide developers that concretely show them how to write code with just enough best practice hand-rails to produce code that makes sense. Creative writing and literature is probably not a good analogy, but more structured styles of writing such as journalism or technical writing which provide loose guidelines may directly relate to software development.


Do common writing styles apply to software developers as well?

There are many styles of writing that aren’t 100% relevant (at first time at least) to software development. For example, persuasive style is aimed at convincing readers of a point of view, but software is not ultimately portraying the opinion of the developer because it is representing the opinion of the users via the developers style.

The workflow styles of writers might be more applicable?

Software can be thought of as three dimensional and non-linear where most literature is more often two dimensional and sequential, so applying structural styles may be more difficult than the process of writing.

Maragaret Atwood described herself as a downhill skier by rushing through 50–60 pages of handwritten pages before “hiking uphill” to revise and begin structuring.
