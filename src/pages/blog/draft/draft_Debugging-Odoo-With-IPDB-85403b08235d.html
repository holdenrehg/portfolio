Debugging software applications can be tricky. Often, I feel more like I’m a detective than I do a developer. Tracing down leads, clues, and messages to their sources sometimes just to be sent off again down another lead until I find the root of the problem.

Tools of the trade

Developers use a big set of different tools for finding the root of these problems:

  • Logger Statements
  • Stack Traces
  • Automated Tests
  • Error Aggregators (Ex. Sentry)
  • Debuggers
  • Educated Guesses (Hypothesize, Experiment, Gather Results, Repeat)

What we’re used to

Most python and Odoo developers seem to start with the _logger method.

  1. Notice a problem
  2. Place _logger.debug(...) around suspected areas
  3. Restart
  4. Check logs
  5. Repeat

This gets you by, but often feels like a huge waste of time especially in systems like Odoo where a restart is required to regenerate .pyc files. Even when using watchers and automatic reloading, it takes some time to go through this process.

The better alternative is to let logger statements do what they do best. Log information to files or external systems for reference, primarily in staging or production environments.

Then pick up a debugger like ipdb to do what it does best. Debug code.
